# 分类模型

分类建模由场景模型、方法模型和生命周期模型组成。

## 场景模型

| 场景 | 适合结构 |
|---|---|
| 个人电脑 | inbox / documents / media / software / archive / backup |
| 学习资料 | projects / resources / knowledge / archive |
| 开发者工作区 | projects / areas / resources / software / archive |
| 家庭/NAS | documents / media / resources / archive / backup |
| 混合磁盘 | inbox / projects / areas / resources / knowledge / documents / media / software / archive / backup |

## 存储拓扑模型

当用户主动提到或实际拥有多个存储介质时，可先判断每个介质的角色，再设计目录。不要把 NAS、移动硬盘、云盘或代码托管平台作为默认必备条件；它们只是可选能力。

常见角色：

| 角色 | 典型介质 | 适合内容 | 不适合内容 |
|---|---|---|---|
| 工作层 / 热数据 | 本机磁盘、SSD、工作区 | 当前项目、当前学习、正在编辑的文档和笔记 | 长期冷归档、重复备份 |
| 在线资料库 / 温数据 | NAS、家庭服务器、云盘 | 电子书、历史课程、照片视频副本、软件仓库、归档资料 | 活跃程序目录、运行中虚拟磁盘、未完成下载 |
| 离线灾备 / 冷数据 | 移动硬盘、离线硬盘 | 照片、视频、重要文档、关键快照 | 临时下载、缓存、可重新获取资源 |
| 版本控制层 | Git 远端、代码托管平台 | 源码、配置、文本型项目历史 | 大型媒体、隐私文档、构建缓存 |
| 同步层 | 笔记同步、WebDAV、云同步客户端 | 需要多设备访问的小文件和知识库 | 频繁变化的大二进制文件 |

可选建模顺序：

1. 先确认用户实际有哪些存储介质或同步工具。
2. 确认每类资料的主副本在哪里。
3. 如用户具备相应条件，再确认是否需要在线副本、离线副本或版本控制。
4. 确认同步方向，优先单向同步，避免双主库冲突。
5. 再生成目录结构、迁移计划和备份计划。

口诀：

```text
本机负责效率
如果有 NAS/云盘，可负责长期可访问
如果有移动硬盘，可负责灾备
如果有代码托管，可负责源码版本
如果有同步工具，可负责多设备访问
```

## 方法模型

### IRK

用来判断内容的价值密度和处理方式：

- Information：过程性材料，跟随任务或项目。
- Resource：可复用资料，进入资源库。
- Knowledge：提炼后的知识，进入知识库。

### PARA

用来判断行动状态：

- Projects：当前项目。
- Areas：长期责任领域。
- Resources：未来可能有用的资源。
- Archives：已完成或不再活跃的内容。

### 时间模型

用来组织归档、凭证、照片、会议资料、学习批次和备份快照。

常用格式：

```text
YYYY
YYYY-MM
YYYY-MM-DD_topic
YYYY-Q1_topic
```

## 生命周期模型

| 状态 | 处理 |
|---|---|
| 收件箱 | 暂存新文件，定期清理 |
| 使用中 | 保留在项目、领域或当前工作目录 |
| 已完成 | 进入归档判断 |
| 归档 | 保留未来查询 |
| 备份 | 作为恢复副本 |
| 删除候选 | 进入待确认清单 |

## 推荐组合

| 使用区域 | 推荐组合 |
|---|---|
| 整个磁盘 | 场景模型 + 生命周期模型 |
| 工作区 | PARA Projects + Archives |
| 笔记库 | IRK Knowledge + PARA Areas |
| 资料库 | IRK Resources + 主题分类 |
| 报销合同 | 时间模型 + documents |
| 照片视频 | 时间模型 + media |
| 备份 | 时间快照 + backup |
| 多存储体系 | 存储拓扑模型 + 生命周期模型 + 备份等级 |
