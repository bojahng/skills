# 文件夹整理规则参考

设计长期可维护的文件夹体系时使用本参考。

## 分类顺序

遇到文件或文件夹时，按顺序判断：

1. 是否还没处理
2. 是否属于某个正在进行的项目
3. 是否是正式凭证或个人重要资料
4. 主要用途是什么
5. 当前处于什么生命周期
6. 是主文件、副本、导出件、缓存还是备份
7. 是否涉及隐私、合同、财务、身份或唯一资料

## IRK 模型

IRK 用来判断内容的性质和处理方式。

| 层级 | 含义 | 特点 | 管理方式 |
|---|---|---|---|
| Information / 信息 | 任务、事件、项目中的过程材料 | 时效性强，复用性低 | 跟随项目或事件，完成后归档 |
| Resource / 资源 | 可反复查阅或复用的外部材料 | 复用性高，可能冗余 | 放入资源库，按主题或类型组织 |
| Knowledge / 知识 | 从信息和资源中提炼后的个人知识 | 价值密度高，冗余低 | 放入知识库或笔记系统 |

判断示例：

```text
项目会议纪要      -> Information
电子书/规范/教程  -> Resource
学习总结/知识卡片 -> Knowledge
```

## PARA 模型

PARA 用来判断行动状态。

| 层级 | 含义 | 适合内容 |
|---|---|---|
| Projects | 有明确目标和结束点的项目 | 当前项目、开发任务、写作任务 |
| Areas | 长期维护的责任领域 | 健康、财务、职业、家庭、运维 |
| Resources | 将来可能有用的资料 | 参考资料、教程、素材、模板 |
| Archives | 已完成或不再活跃的内容 | 旧项目、旧课程、历史材料 |

## 时间模型

时间模型用于归档、批次、凭证和快照，不建议替代全部分类。

推荐格式：

```text
YYYY
YYYY-MM
YYYY-MM-DD_主题
YYYY-Q1_主题
YYYY_主题
```

适合：

- 报销
- 合同
- 照片
- 项目归档
- 学习批次
- 会议资料
- 备份快照

不适合：

- 当前项目主目录
- 常用软件目录
- 长期知识库顶层
- 需要按主题查找的资源库

## 默认顶层目录

默认采用“编号 + 英文”，并附中文说明：

| 目录 | 中文名称 | 用途 |
|---|---|---|
| `00.inbox` | 收件箱 | 新下载、新接收、待分类文件 |
| `01.projects` | 项目 | 正在推进、有明确完成目标的事项 |
| `02.areas` | 领域 | 长期维护的责任和主题 |
| `03.resources` | 资源 | 可复用资料、教程、规范、素材 |
| `04.knowledge` | 知识 | 沉淀后的笔记、总结、知识卡片 |
| `10.documents` | 文档 | 合同、报销、证明、简历、表格 |
| `20.media` | 媒体 | 照片、视频、录屏、素材 |
| `30.software` | 软件 | 安装包、驱动、工具、绿色软件 |
| `90.archive` | 归档 | 已完成但未来可能查询的资料 |
| `99.backup` | 备份 | 用于恢复的副本 |

## 目录说明文件

顶级目录和次级目录默认都要有一个 `目录说明.txt`。

说明文件建议包含：

- 目录名称
- 中文用途
- 放什么
- 不放什么
- 维护周期
- 风险和备份提醒

适用范围：

- 磁盘顶层的受管理目录，例如 `01.study`、`04.software`、`12.book`。
- 受管理目录下的次级分类目录，例如 `04.software/01.os`、`12.book/02.programming`。

可跳过的目录：

- Git 仓库或项目子目录，避免产生无关未跟踪文件。
- 系统目录、应用安装目录、虚拟磁盘目录。
- Obsidian/WebDAV 等同步工具强绑定目录，除非用户明确确认。
- 临时缓存目录、未完成下载目录。

真实执行整理后，应创建或更新这些说明文件；如果跳过，执行日志中记录原因。

## 生命周期模型

推荐生命周期：

```text
收件箱 -> 使用中 -> 已完成 -> 归档 / 备份 / 删除候选
```

含义：

- 收件箱：新下载、新接收、用途不明的文件。
- 使用中：仍在编辑、学习、开发、制作或查阅的文件。
- 已完成：当前不再频繁变化的文件。
- 归档：完成后保留以便未来查询。
- 备份：为了恢复而保存的副本。
- 删除候选：没有实际保留价值，但删除前必须确认。

## 判断口诀

```text
还没处理？              -> 00.inbox
正在做项目？            -> 01.projects
长期责任或主题？        -> 02.areas
可复用外部资料？        -> 03.resources
已经整理成知识？        -> 04.knowledge
合同、报销、证明？      -> 10.documents
照片、视频、录屏？      -> 20.media
软件、工具、驱动？      -> 30.software
已完成但还要查？        -> 90.archive
只是为了防丢？          -> 99.backup
没有价值？              -> 删除候选
```

## 命名规则

目录默认使用编号 + 英文。文件和子目录按场景使用日期、主题、版本。

推荐：

```text
00.inbox
03.note
02.system_tools
YYYY-MM-DD_topic_description
YYYY_topic
topic_v1
topic_purpose
```

目录命名补充：

- 编号目录默认使用两位数字前缀。
- 数字和英文之间使用点号，例如 `03.note`。
- 多词英文目录使用下划线，例如 `02.system_tools`。
- 避免空格、短横线、大小写混排和含义过窄的扩展名目录。

避免：

```text
新建文件夹
临时
杂项
资料
最终
最终最终
最新版
不知道
03.Note
02.system-tools
```

版本命名推荐：

```text
2026-05-26_resume_v1.docx
2026-06-10_resume_submission.docx
```

## 归档和备份的区别

归档：

- 可以是主文件完成后的最终存放地
- 目的是未来查询
- 应该能被搜索和阅读
- 可以按年份、主题或项目组织

备份：

- 是为了恢复的副本
- 不是日常编辑入口
- 应该标明来源或备份日期
- 不应该保存唯一一份重要资料

## 多存储规则

当用户同时拥有或主动提到本机磁盘、NAS、移动硬盘、云盘或代码托管平台时，可以先设计存储关系，再设计目录。不要把这些介质作为默认强制选项；如果用户只有一个磁盘，就按单盘规则设计。

### 角色划分

| 角色 | 放什么 | 不放什么 |
|---|---|---|
| 工作盘 | 当前项目、当前学习、正在编辑的笔记、下载入口 | 冷备份、完整历史库、重复副本 |
| NAS / 在线资料库 | 电子书、旧课程、照片视频副本、软件仓库、长期归档 | 活跃程序目录、运行中虚拟磁盘、未完成下载 |
| 移动硬盘 / 离线备份 | 不可替代资料、照片、视频、重要文档、关键快照 | 临时下载、缓存、可重新获取资源 |
| 代码托管 | 源码、配置、文本型项目历史 | 大型媒体、隐私资料、构建产物 |
| 笔记同步 | 知识库、小文件、多设备访问内容 | 原始课程大包、安装包、视频合集 |

### 主副本规则

建议尽量明确每类资料的主副本位置：

1. 活跃项目通常以本机为工作副本；如果用户使用代码托管，可作为版本备份。
2. 笔记库通常以本机或同步目录为主编辑入口；如果用户有同步工具，可增加在线同步副本和离线快照。
3. 电子书、旧课程、软件仓库适合放在长期资料库；资料库可以是 NAS、外置硬盘、云盘或本机归档区。
4. 照片、视频、合同、报销、证件等高价值资料建议至少两份；是否三份取决于用户可用介质和风险偏好。
5. 未完成下载、缓存、日志、构建产物通常不进入长期备份体系。

### 同步方向规则

如果存在多个存储介质或同步工具，优先单向同步，避免双向冲突：

| 内容 | 推荐方向 |
|---|---|
| 当前项目 | 本机 -> 代码托管 |
| 笔记 | 主编辑端 -> 同步端 -> 离线快照 |
| 旧课程 | 本机 -> NAS/归档 |
| 软件包 | 本机整理后 -> NAS/软件仓库 |
| 照片视频 | 主库 -> 在线副本 -> 离线副本 |
| 正式文档 | 主库 -> 在线副本 -> 离线副本 |

### 3-2-1 备份原则

对不可替代资料，可把 3-2-1 作为理想目标，而不是强制要求：

```text
3 份数据
2 种介质
1 份离线或异地
```

不要求所有资料都三份，也不要因为用户没有足够介质而阻塞整理。可重新下载、可重新生成、低价值缓存不进入高等级备份。

## 沙盒预览规则

沙盒整理的目标是预览结构，不是立刻制造第二份资料库。

默认策略：

- 小文件可以复制到沙盒，方便用户直接查看。
- 大文件只进入待确认清单，避免重复占用空间。
- 整个文件夹只进入待确认清单，避免递归复制大量内容。
- 照片、家庭视频、合同、报销、项目源文件等高价值资料只做清单，不做自动复制。
- 沙盒说明必须明确哪些内容只是建议迁移项，避免用户误以为已经完成真实迁移。

正式实施时：

1. 提醒用户先备份重要数据。
2. 确认用户已经备份或接受风险。
3. 重新生成迁移计划。
4. 明确每个清单项的最终动作：复制、移动、归档、保留原地或跳过。
5. 对大目录和高价值资料分批确认。

## 冲突处理规则

当文件可以放到多个地方时：

1. 如果属于正在进行的项目，优先放项目目录。
2. 如果是合同、财务、身份、证明等正式记录，优先放文档区。
3. 如果是软件、驱动或工具，优先放软件区。
4. 如果是沉淀后的知识，优先放知识区。
5. 如果只是原始参考资料，优先放资源区。
6. 如果不确定，放收件箱等待分类。

## 删除规则

可以作为删除候选：

- 重复下载
- 临时解压目录
- 明确无用的缓存
- 已确认过期的安装包
- 已归档且有备份的临时副本

谨慎处理：

- 个人资料
- 报销凭证
- 合同证明
- 原始照片
- 项目源文件
- 唯一副本

删除前检查：

```text
是否唯一？
是否重要？
是否已有备份？
未来是否可能查？
```

## 重复文件规则

| 情况 | 处理 |
|---|---|
| 完全相同的下载副本 | 删除多余副本前先确认 |
| 一个是主文件，一个是备份 | 保留主文件和备份 |
| 一个是旧版本，一个是新版本 | 保留必要版本，删除无意义版本 |
| 不确定哪个重要 | 放入待确认清单 |

## 维护周期

推荐节奏：

- 每周：清理收件箱和下载目录。
- 每月：检查工作区和已完成项目。
- 每季度：检查备份、大文件、重复文件和归档。
