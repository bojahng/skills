# 文件夹整理规则参考

设计长期可维护的文件夹体系时使用本参考。

## 分类顺序

遇到文件或文件夹时，按顺序判断：

1. 是否还没处理
2. 是否属于某个正在进行的项目
3. 是否是正式凭证或个人重要资料
4. 主要用途是什么
5. 当前处于什么生命周期
6. 是主文件、副本、导出件、缓存还是备份
7. 是否涉及隐私、合同、财务、身份或唯一资料

## IRK 模型

IRK 用来判断内容的性质和处理方式。

| 层级 | 含义 | 特点 | 管理方式 |
|---|---|---|---|
| Information / 信息 | 任务、事件、项目中的过程材料 | 时效性强，复用性低 | 跟随项目或事件，完成后归档 |
| Resource / 资源 | 可反复查阅或复用的外部材料 | 复用性高，可能冗余 | 放入资源库，按主题或类型组织 |
| Knowledge / 知识 | 从信息和资源中提炼后的个人知识 | 价值密度高，冗余低 | 放入知识库或笔记系统 |

判断示例：

```text
项目会议纪要      -> Information
电子书/规范/教程  -> Resource
学习总结/知识卡片 -> Knowledge
```

## PARA 模型

PARA 用来判断行动状态。

| 层级 | 含义 | 适合内容 |
|---|---|---|
| Projects | 有明确目标和结束点的项目 | 当前项目、开发任务、写作任务 |
| Areas | 长期维护的责任领域 | 健康、财务、职业、家庭、运维 |
| Resources | 将来可能有用的资料 | 参考资料、教程、素材、模板 |
| Archives | 已完成或不再活跃的内容 | 旧项目、旧课程、历史材料 |

## 时间模型

时间模型用于归档、批次、凭证和快照，不建议替代全部分类。

推荐格式：

```text
YYYY
YYYY-MM
YYYY-MM-DD_主题
YYYY-Q1_主题
YYYY_主题
```

适合：

- 报销
- 合同
- 照片
- 项目归档
- 学习批次
- 会议资料
- 备份快照

不适合：

- 当前项目主目录
- 常用软件目录
- 长期知识库顶层
- 需要按主题查找的资源库

## 默认顶层目录

默认采用“编号 + 英文”，并附中文说明：

| 目录 | 中文名称 | 用途 |
|---|---|---|
| `00.inbox` | 收件箱 | 新下载、新接收、待分类文件 |
| `01.projects` | 项目 | 正在推进、有明确完成目标的事项 |
| `02.areas` | 领域 | 长期维护的责任和主题 |
| `03.resources` | 资源 | 可复用资料、教程、规范、素材 |
| `04.knowledge` | 知识 | 沉淀后的笔记、总结、知识卡片 |
| `10.documents` | 文档 | 合同、报销、证明、简历、表格 |
| `20.media` | 媒体 | 照片、视频、录屏、素材 |
| `30.software` | 软件 | 安装包、驱动、工具、绿色软件 |
| `90.archive` | 归档 | 已完成但未来可能查询的资料 |
| `99.backup` | 备份 | 用于恢复的副本 |

## 生命周期模型

推荐生命周期：

```text
收件箱 -> 使用中 -> 已完成 -> 归档 / 备份 / 删除候选
```

含义：

- 收件箱：新下载、新接收、用途不明的文件。
- 使用中：仍在编辑、学习、开发、制作或查阅的文件。
- 已完成：当前不再频繁变化的文件。
- 归档：完成后保留以便未来查询。
- 备份：为了恢复而保存的副本。
- 删除候选：没有实际保留价值，但删除前必须确认。

## 判断口诀

```text
还没处理？              -> 00.inbox
正在做项目？            -> 01.projects
长期责任或主题？        -> 02.areas
可复用外部资料？        -> 03.resources
已经整理成知识？        -> 04.knowledge
合同、报销、证明？      -> 10.documents
照片、视频、录屏？      -> 20.media
软件、工具、驱动？      -> 30.software
已完成但还要查？        -> 90.archive
只是为了防丢？          -> 99.backup
没有价值？              -> 删除候选
```

## 命名规则

目录默认使用编号 + 英文。文件和子目录按场景使用日期、主题、版本。

推荐：

```text
YYYY-MM-DD_topic_description
YYYY_topic
topic_v1
topic_purpose
```

避免：

```text
新建文件夹
临时
杂项
资料
最终
最终最终
最新版
不知道
```

版本命名推荐：

```text
2026-05-26_resume_v1.docx
2026-06-10_resume_submission.docx
```

## 归档和备份的区别

归档：

- 可以是主文件完成后的最终存放地
- 目的是未来查询
- 应该能被搜索和阅读
- 可以按年份、主题或项目组织

备份：

- 是为了恢复的副本
- 不是日常编辑入口
- 应该标明来源或备份日期
- 不应该保存唯一一份重要资料

## 沙盒预览规则

沙盒整理的目标是预览结构，不是立刻制造第二份资料库。

默认策略：

- 小文件可以复制到沙盒，方便用户直接查看。
- 大文件只进入待确认清单，避免重复占用空间。
- 整个文件夹只进入待确认清单，避免递归复制大量内容。
- 照片、家庭视频、合同、报销、项目源文件等高价值资料只做清单，不做自动复制。
- 沙盒说明必须明确哪些内容只是建议迁移项，避免用户误以为已经完成真实迁移。

正式实施时：

1. 提醒用户先备份重要数据。
2. 确认用户已经备份或接受风险。
3. 重新生成迁移计划。
4. 明确每个清单项的最终动作：复制、移动、归档、保留原地或跳过。
5. 对大目录和高价值资料分批确认。

## 冲突处理规则

当文件可以放到多个地方时：

1. 如果属于正在进行的项目，优先放项目目录。
2. 如果是合同、财务、身份、证明等正式记录，优先放文档区。
3. 如果是软件、驱动或工具，优先放软件区。
4. 如果是沉淀后的知识，优先放知识区。
5. 如果只是原始参考资料，优先放资源区。
6. 如果不确定，放收件箱等待分类。

## 删除规则

可以作为删除候选：

- 重复下载
- 临时解压目录
- 明确无用的缓存
- 已确认过期的安装包
- 已归档且有备份的临时副本

谨慎处理：

- 个人资料
- 报销凭证
- 合同证明
- 原始照片
- 项目源文件
- 唯一副本

删除前检查：

```text
是否唯一？
是否重要？
是否已有备份？
未来是否可能查？
```

## 重复文件规则

| 情况 | 处理 |
|---|---|
| 完全相同的下载副本 | 删除多余副本前先确认 |
| 一个是主文件，一个是备份 | 保留主文件和备份 |
| 一个是旧版本，一个是新版本 | 保留必要版本，删除无意义版本 |
| 不确定哪个重要 | 放入待确认清单 |

## 维护周期

推荐节奏：

- 每周：清理收件箱和下载目录。
- 每月：检查工作区和已完成项目。
- 每季度：检查备份、大文件、重复文件和归档。
